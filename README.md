# Поиск статей справки Авито по вопросу пользователя

Решение задачи ранжирования: по тексту вопроса в поддержку вернуть топ-10 статей справки,
отсортированных по релевантности. Это retrieval-этап RAG-пайплайна - генерации ответа
в задаче нет, метрика качества - **MAP@10**.

**Результат на публичном тесте: MAP@10 = 0.64.**

## Архитектура решения

Классическая многоступенчатая схема «от дешёвого к точному»: каждая следующая ступень
дороже предыдущей, поэтому работает со всё меньшим числом кандидатов.

```text
запрос
  │
  ├─ BM25 (лексический поиск: заголовок ×0.5 + тело)          ┐
  ├─ dense-поиск BAAI/bge-m3 по чанкам 384 токена,            ├─ fusion = 0.75·dense + 0.25·bm25
  │  скор статьи = среднее 2 лучших чанков                    ┘        │
  │                                                              топ-50 кандидатов
  │                                                                    │
  ├─ cross-encoder BAAI/bge-reranker-v2-m3                             │
  │  читает пары «запрос + 2 ближайших чанка кандидата»    ────────────┤
  │                                                                    │
  └─ LambdaMART (LightGBM) на 62 признаках ступеней выше   - итоговый порядок → топ-10
```

Качество по ступеням на локальной валидации (development, 400 запросов):

| Ступень | MAP@10 |
|---|---|
| BM25 | 0.256 |
| Dense-поиск по чанкам | 0.495 |
| Fusion | 0.501 |
| + cross-encoder (zero-shot) | 0.555 |
| + LambdaMART (OOF-оценка) | 0.682 |

Oracle recall@50 кандидатной ступени - 0.97: почти все правильные статьи попадают в топ-50,
поэтому вся дальнейшая работа - это ранжирование внутри полусотни.

## Структура репозитория

```text
notebooks/
├── 01_data_preprocessing.ipynb       - анализ данных, очистка HTML, чанкинг
├── 02_retrieval_and_candidates.ipynb - BM25, эмбеддинги, fusion, топ-50 кандидатов
├── 03_reranking.ipynb                - скоринг cross-encoder'ом
└── 04_final_ranking_and_answer.ipynb - LambdaMART, анализ ошибок, answer.csv
requirements.txt
README.md
```

Каждый ноутбук самодостаточен по объяснениям: все термины (retrieval, эмбеддинги,
cross-encoder, learning-to-rank, OOF и т.д.) вводятся по-русски там, где впервые нужны.

## Как воспроизвести

1. Создайте на Google Диске папку `avito_rag/data` и положите в неё три файла задачи:
   `articles.f`, `calibration.f`, `test.f`. Больше ничего готовить не нужно - все
   промежуточные папки и файлы ноутбуки создают на Диске сами.
2. Откройте ноутбуки в Google Colab и выполните их по порядку (в каждом - Runtime -> Run all):

   | Ноутбук | Runtime | Время с нуля |
   |---|---|---|
   | 01 | CPU | ~10–15 мин |
   | 02 | **T4 GPU** | ~20–30 мин |
   | 03 | **T4 GPU** | ~40–60 мин |
   | 04 | CPU | ~10–15 мин |

3. Итоговый файл появится по пути `avito_rag/answer.csv` (ноутбук 04 также скачает его
   автоматически и проверит формат: 500 строк, по 10 уникальных существующих `article_id`).

Повторный запуск быстрый: тяжёлые артефакты (эмбеддинги, скоры cross-encoder'а)
кэшируются на Диске с контрольными SHA-256-хэшами входов и пересчитываются только если
входные данные изменились.

### Замечания о воспроизводимости

- Все случайности зафиксированы (`seed 42`): разбиение development/holdout, фолды
  кросс-валидации, LightGBM.
- Ревизия весов cross-encoder'а фиксируется в метаданных при первом запуске и
  переиспользуется при последующих.
- Единственный источник недетерминизма - вычисления нейросетей на GPU в float16:
  при пересчёте эмбеддингов/логитов на другом железе значения могут отличаться в последних
  знаках и в редких случаях менять местами кандидатов с почти равными скорами. На итоговую
  метрику это влияет в пределах тысячных.

## Описание подхода

### Как обрабатывался HTML статей

Тела статей — HTML с таблицами, вкладками, спойлерами. Я разбираю его BeautifulSoup (lxml)
по принципу «сохранить всё, что видит пользователь, убрать всё служебное»:

- удаляются только технические теги без видимого текста: `script`, `style`, `noscript`, `svg`, `input`, `iframe`;
- заголовки и абзацы становятся отдельными строками, пункты списков получают маркер `- `,
  строки таблиц соединяются через ` | `, текст вкладок и спойлеров сохраняется;
- сохраняется видимый текст ссылок, но не URL из `href` (шум для поиска);
- `alt`-подписи картинок проходят фильтр (с декодированием percent-encoding): осмысленные
  описания интерфейса остаются, имена файлов и URL выбрасываются;
- дубль заголовка в первой строке тела убирается.

Качество очистки контролировалось глазами на разных типах статей и автоматической
диагностикой по корпусу (пустые результаты, остатки тегов, аномальное сжатие текста).

Для векторного поиска очищенные статьи режутся на **чанки по 384 токена с перекрытием 58**
токенизатором bge-m3; в каждый чанк добавляется префикс с заголовком статьи, чтобы фрагмент
из середины не терял контекст. Окна строятся по токенам (кодирование -> разрезание списка
токенов -> декодирование), что гарантирует точный размер и перекрытие. Размер чанка выбран
сравнением вариантов по MAP@10 на development: чанки заметно обыгрывают кодирование статьи
целиком (0.49 против 0.40), потому что вектор длинного документа «размывается».

### Какие признаки, модели и алгоритмы использовались

**Ступень 1 — кандидаты.** BM25 (rank-bm25) отдельно по заголовкам (вес 0.5) и телам;
dense-поиск `BAAI/bge-m3` по чанкам, скор статьи - среднее двух лучших чанков (top2-mean:
устойчивее максимума к случайным выбросам и не топит длинные статьи, где релевантен один
раздел). Скоры нормализуются per-query и складываются: `0.75·dense + 0.25·bm25`. Все веса
выбраны перебором на development. Берётся топ-50.

**Ступень 2 — реранкер.** Cross-encoder `BAAI/bge-reranker-v2-m3` zero-shot (без дообучения)
скорит пары «запрос + чанк» для двух ближайших к запросу чанков каждого кандидата.
В отличие от эмбеддера, он читает запрос и текст совместно, что даёт качественно другой
уровень точности на перефразировках.

**Ступень 3 — learning-to-rank.** LambdaMART (LightGBM `LGBMRanker`, lambdarank с обрезкой
на топ-10 — ровно под метрику) на 62 признаках пары «запрос–статья»: скоры и ранги всех
предыдущих ступеней, их per-query нормировки (min-max, z-score), обратные/логарифмированные
ранги, агрегаты логитов реранкера по чанкам (max/mean/min/std/range), векторные близости
чанков и лексические пересечения запроса с заголовком и телом (общие слова, Жаккар, точные
вхождения). Неглубокие деревья и сильная регуляризация - данных всего 400 обучающих запросов.

### Как проверялось качество на calibration.f

- Один раз и навсегда: разбиение 500 запросов на **development (400)** и **holdout (100)** -
  группами по нормализованному тексту запроса (дубликаты не расползаются по частям)
  со стратификацией по числу релевантных статей.
- Все выборы конфигураций - только по development; для LTR - **5-фолдовая out-of-fold
  кросс-валидация** (фолды тоже группами и со стратификацией), ранняя остановка по фолду,
  итоговое число деревьев - медиана по фолдам.
- Holdout использовался как одноразовая финальная проверка уже принятых решений.
- Для test модель переобучается на всех 500 размеченных запросах с зафиксированными
  гиперпараметрами.

### Какие типы ошибок нашлись при анализе и что с ними сделано

1. **Путаница тематически близких статей** (доставка/возврат для покупателя vs продавца,
   разные сервисы с похожими правилами). Векторный поиск их почти не различает. Ответ -
   cross-encoder, который читает формулировки целиком, и лексические признаки точных
   совпадений с заголовком.
2. **Многоответные запросы**: AP@10 систематически ниже при 3+ релевантных статьях.
   Ответ — оптимизация именно MAP (lambdarank/труncation 10), а не первой позиции,
   и запас кандидатов (топ-50 при oracle recall 0.97).
3. **Короткие и разговорные запросы с опечатками** («не приходят деньги») - мало лексических
   сигналов. Ответ — нормализация текста (юникод, пунктуация, плейсхолдеры) и опора на
   векторную составляющую fusion.
4. **Переобучение на частоты статей** - главный найденный риск. Промежуточная версия
   с признаками-приорами (как часто статья была правильным ответом в calibration) давала
   +0.02 MAP@10 локально и **−0.06 на публичном тесте**: распределение «популярных» статей
   в тестовых запросах другое, а через локальную валидацию такой признак протекает
   (он посчитан по всему calibration сразу). Решение - полный отказ от любых признаков
   популярности статей; в модели только признаки пары «запрос–статья». После этого разрыв
   локальной и тестовой метрики сократился до ожидаемого статистического (~0.04 при OOF 0.68 -> тест 0.64).
5. **Гигантская статья на десятки тысяч токенов** (сотни чанков) - риск смещения
   chunk-level поиска в её сторону. Решение общее, без спецобработки конкретной статьи:
   агрегация top2-mean, при которой количество чанков само по себе преимущества не даёт.

### Что ещё проверялось и было отклонено

- TF-IDF (словный, символьный, комбинированный) - слабее BM25 на development.
- Кодирование статей целиком и «заголовок + чанки» - слабее чистых чанков top2-mean.
- Бленды LTR-скора с zero-shot реранкером (75/25, 50/50) - хуже чистого LTR и локально,
  и на тесте.
- Ансамбль LightGBM по 5 сидам - без изменений локально, на тесте -0.01: дисперсия
  LTR-ступени не является узким местом.

## Ограничения задачи

Внешние LLM/API не использовались: все модели (bge-m3, bge-reranker-v2-m3, LightGBM) -
локальные open-source, тестовые запросы никуда не отправляются. Ручной разметки test
и подгонки под конкретные `query_id` нет - пайплайн полностью общий.
